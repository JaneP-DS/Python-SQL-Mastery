# 📚 SQLD 제1과목: 데이터 모델링의 이해 정리

---

## 📊1. 데이터 모델의 이해

### § 데이터 모델링 정의
현실 세계의 복잡한 비즈니스 프로세스를 추상화, 단순화, 명확화하여 중요한 요소와 관계를 일정한 표기법에 의해 표현하는 것.

* **추상화(Abstraction):** 현실의 복잡한 데이터를 중요한 정보만 남기고 표현하는 과정.(=모형화)
* **단순화(Simplification):** 복잡한 데이터를 이해하기 쉬운 구조로 표현하는 과정.
* **명확화(Clarification):** 데이터의 구조와 관계를 명확하게 표현하여 오해의 여지를 없애는 과정.

### § 데이터 모델링의 3단계
1. **개념적 데이터 모델링:** 추상화 수준이 가장 높으며, 업무 중심적인 포괄적 모델링 방식. (ERD 도출).
2. **논리적 데이터 모델링:** 논리적 단계와 데이터 속성을 정의.(정규화 및 반정규화 수행, 기본 키 외래 키 지정, 테이블 도출).
3. **물리적 데이터 모델링:** 실제 DBMS에 맞는 테이블 구축.(데이터 타입 정의, 인덱스 설계, 뷰 설계).


# 🗂️ 데이터베이스 스키마 (Schema) 이해하기

데이터베이스의 구조와 제약 조건에 관한 전반적인 명세를 정의한 것.

---

## 1. 스키마의 정의
스키마는 데이터베이스를 구성하는 **데이터 개체(Entity), 속성(Attribute), 관계(Relationship)** 및 데이터 조작 시 준수해야 할 **제약 조건**을 정의한 것.

* **정적 특징:** 데이터베이스에 저장된 값(인스턴스)은 수시로 변하지만, 스키마 자체는 한 번 정의되면 자주 변경되지 않음.

---

## 2. 스키마와 인스턴스 비교
헷갈리기 쉬운 두 개념의 차이점.

* **스키마 (Schema):** 데이터베이스의 **틀 (구조)**. (예: 테이블의 컬럼명, 데이터 타입)
* **인스턴스 (Instance):** 특정 시점에 스키마에 따라 저장된 **실제 데이터 값**. (예: 테이블에 들어있는 레코드들)

---
### § 데이터베이스의 3단계 구조

1. **외부 스키마:** 응용 프로그램이나 사용자 관점에서 데이터를 표현한 것.
2. **개념 스키마:** 외부스키마에서 요구된 모든 데이터를 통합하여 설계한 데이터의 논리적 구조를 나타냄. 외부 스키마가 필요로 하는 데이터를 모아 데이터 간의 관계를 정의.(데이터 모델링은 통합 관점의 개념 스키마를 만들어가는 과정.)
3. **내부 스키마:** 개념스키마를 물리적으로 저장하는 구조. 데이터가 실제로 저장되는 방식(파일, 인덱스 등)과 최적화된 데이터베이스 구조를 포함함.

#### ※데이터 독립성: 하위 스키마가 변경되어도 상위 스키마에 영향을 주지 않는 특성.
---

## 📊2. 엔터티 (Entity)

업무에서 관리하고자 하는 대상.(사물, 사건, 개념 등)


## 엔터티의 분류 (Entity Classification)

### 1. 유무형에 따른 분류 (물리적 형태 여부)
* **유형 엔터티 (Tangible):** 물리적 형태가 있고 지속적으로 활용되는 엔터티 (예: 사원, 상품, 강사).
* **개념 엔터티 (Conceptual):** 물리적 형태는 없으나 개념적으로 존재하는 엔터티 (예: 부서, 학과, 보험상품).
* **사건 엔터티 (Event):** 업무 수행 과정에서 발생하는 엔터티로, 통계나 이력 관리에 주로 사용됨 (예: 주문, 결제, 수강신청).

### 2. 발생 시점에 따른 분류 (중요도에 따른 분류)
| 분류 | 설명 | 예시 |
| :--- | :--- | :--- |
| **기본/키 엔터티 (Fundamental)** | 다른 엔터티로부터 영향을 받지 않고 독립적으로 생성됨. 스스로 주식별자를 가짐. | 사원, 고객, 상품, 부서 |
| **중심 엔터티 (Main)** | 기본 엔터티로부터 발생하며, 업무의 중심 역할을 함 | 주문, 접수, 계약 |
| **행위 엔터티 (Active)** | 두 개 이상의 엔터티로부터 발생하며 내용이 자주 변경됨 | 주문내역, 배송내역 |

---
> **엔터티의 성립 조건**
> - 반드시 해당 업무에서 필요하고 관리해야 하는 정보여야 함.
> - **유일한 식별자**에 의해 식별이 가능해야 함.
> - 2개 이상의 **인스턴스(Instance)**의 집합이어야 함.
> - 반드시 **속성(Attribute)**을 가져야 함.
> - 다른 엔터티와 최소 1개 이상의 **관계**가 있어야 함.

---

## 📊3. 속성 (Attribute)

엔터티를 구성하는 가장 기본적인 정보 단위.

* **분류 (특성에 따라):**
    * **기본 속성:** 업무로부터 추출된 일반적인 속성.
    * **설계 속성:** 업무에는 없으나 모델링 편의를 위해 만든 속성 (예: 코드, 일련번호).
    * **파생 속성:** 타 속성의 영향을 받아 계산된 속성 (예: 합계, 평균).

---

## 📊4. 관계 (Relationship)

엔터티간의 연결 또는 상호작용.

## § 관계의 분류
**• 존재에 의한 관계:** 존재 자체로 엔터티 간 연관성이 형성되는 관계를 나타냄.

**• 행위에 의한 관계:** 특정한 업무 수행이나 이벤트 발생을 통해 엔터티 간 연관성이 형성되는 관계를 나타냄.

* **관계명:** 관계의 이름 (현재형 동사 사용).
* **차수(Cardinality):** 1:1, 1:N, M:N 등 연결되는 개수.
* **선택사양(Optionality):** 필수(Mandatory)인지 선택(Optional)인지 여부.



---

## 📊5. 식별자 (Identifier)

엔터티 내에서 인스턴스를 유일하게 구분할 수 있는 속성.

### 주식별자의 4가지 특징
1.  **유일성:** 엔터티 내 모든 인스턴스를 유일하게 구분.
2.  **최소성:** 유일성을 만족하는 최소한의 속성 수로 구성.
3.  **불변성:** 한 번 특정된 주식별자 값은 변하지 않아야 함.
4.  **존재성:** 주식별자 값은 반드시 존재해야 함 (Not Null).

---

## 📊6. 본질식별자 vs 인조식별자

주식별자를 어떻게 선정하느냐에 따라 구분됨.

| 구분 | 본질식별자 (Natural) | 인조식별자 (Surrogate) |
| :--- | :--- | :--- |
| **정의** | 엔터티의 고유성을 나타내는 본질적인 속성을 식별자로 사용 | 관리 편의를 위해 인위적으로 부여한 식별자 |
| **예시** | 주민등록번호, 학번, 사업자번호 | 주문번호, 일련번호 |
| **장점** | 데이터 고유성 보장, 중복 데이터 방지 | 주식별자 구조 단순화, 쿼리 작성 용이 |
| **단점** | 여러 속성이 묶일 경우 관리와 인덱싱이 어려움 | 데이터 해석에 어려움, 추가적인 인덱스나 제약 조건 필요 |
